# MODULES.md — fintrac-review

Three modules layer on top of the existing intake form to form the full value ladder:

| Module | Route | Status | Purpose |
|--------|-------|--------|---------|
| Intake | `/` | Live | Lead qualification — feeds Hugh |
| Diagnosis | `/diagnosis` | Skeleton | Loss leader — free 5-pillar health check |
| Consultation | `/consultation` | Skeleton | Discuss diagnosis results, scope engagement |
| Program | `/program/[clientId]` | Skeleton | Client portal — remediation tracking + documents |

## Module 1 — Diagnosis (Loss Leader)

Free AML Compliance Health Check tool. Prospect completes a 5-pillar self-assessment, receives an instant scorecard (Green / Amber / Red) with per-pillar narrative generated by Claude. Email-gated. Token-based shareable URL. Integrates with Airtable and GHL for lead capture.

### Backlog

#### Question bank — 5-pillar self-assessment (~20 questions with scoring rubrics)

**What this is:** The core intellectual content of the tool. Matthew's domain expertise encoded into ~20 questions (4–5 per pillar), each with structured answer options and a score weight (0–4 per pillar). The scoring engine in `lib/diagnosis/scoring.ts` is a thin shell until this exists.

**The 5 pillars (PCMLTFA s. 9.6 compliance program requirements):**

| Pillar | Key obligation |
|--------|---------------|
| Policies & Procedures | Written, current, approved, distributed |
| Risk Assessment | Entity-level + relationship-level, documented, reviewed |
| Training Program | Role-based, tracked, frequency meets risk level |
| Ongoing Monitoring | Transaction monitoring, KYC refresh, STR/LCTR review |
| Compliance Officer | Appointed, senior enough, resourced |

**Format needed per question:**
```ts
{
  id: "policies_q1",
  pillar: "policies",
  question: "When were your AML/ATF policies and procedures last reviewed and approved?",
  options: [
    { value: 0, label: "We don't have written policies" },
    { value: 1, label: "More than 3 years ago" },
    { value: 2, label: "1–3 years ago" },
    { value: 3, label: "Within the last year" },
    { value: 4, label: "Within the last year and following a material change to our business or regulatory requirements" },
  ],
  weight: 1.0,
}
```

**Scoring rubric:**
- Each pillar: 4–5 questions × max 4 pts each = max 16–20 pts per pillar
- Normalize to 0–4 per pillar for display
- Overall: sum of 5 pillar scores (0–20)
- Rating thresholds: Strong (≥75%), Gaps Identified (40–74%), Material Deficiencies (<40%)

**Owner:** Matthew Levine — requires his review and approval before implementation.

## Module 2 — Consultation

Structured discussion of diagnosis results. Prospect lands here from the diagnosis CTA (`/consultation?token=[token]`). Booking widget (GHL calendar or Calendly) passes the diagnosis token as metadata. On booking confirmed: Hugh's `call-prep` task fires using diagnosis data; Matthew receives a brief before the call.

### Files (skeleton complete)

| File | Purpose |
|------|---------|
| `app/consultation/page.tsx` | Landing page — reads `?token`, renders context + booking widget |
| `app/api/consultation/book/route.ts` | POST: store booking, fire Hugh call-prep, notify GHL |
| `components/consultation/booking-widget.tsx` | GHL calendar or Calendly embed |

### Backlog

- Decide scheduling tool: GHL calendar embed vs. Calendly (impacts `BookingWidget` implementation)
- Airtable table: `consultations` (id, diagnosis_token, call_date, status, prep_notes)
- Hugh `call-prep` integration: POST to internal endpoint → generate brief → store against consultation record

## Module 3 — Program Design & Remediation

Auth-gated client portal. Created when a consultation converts to a paid engagement. Matthew creates the program (manually or via API), which pre-populates a remediation checklist generated by Claude from the diagnosis report. Client and Matthew both access the portal.

### Files (skeleton complete)

| File | Purpose |
|------|---------|
| `app/program/[clientId]/page.tsx` | Dashboard: status, progress, deadlines |
| `app/program/[clientId]/remediation/page.tsx` | Remediation checklist by pillar |
| `app/program/[clientId]/documents/page.tsx` | Upload/download program documents |
| `app/api/program/[clientId]/route.ts` | GET/PATCH program data |
| `lib/program/types.ts` | Program, RemediationItem, DocumentRecord |
| `lib/program/db.ts` | Supabase query helpers |
| `lib/integrations/supabase.ts` | Supabase client |
| `components/program/*` | Dashboard, RemediationTracker, DocumentManager |

### Backlog

- Supabase project setup: `programs`, `remediation_items`, `documents` tables (SQL migrations)
- Auth: Supabase magic link, middleware protecting `app/program/` routes
- Remediation list generation: Claude generates checklist from diagnosis report on program creation
- Document storage: Supabase Storage, signed URL downloads, indexed by program + pillar
- Program creation flow: Matthew-facing admin action (manual or triggered from consultation record)

### Prerequisite

- Supabase project must be provisioned before any Phase 3 work begins
- `SUPABASE_URL` and `SUPABASE_SERVICE_KEY` env vars required (see `.env.example`)
